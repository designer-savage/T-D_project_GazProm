# Корпоративная экосистема T&D на базе AI-агентов
### ИТ-кластер Газпром Нефти — техническая концепция

---

## Суть системы

Сотрудник открывает портал и пишет что-то вроде:
> «Хочу стать тимлидом через год, что мне нужно прокачать?»

Дальше происходит следующее:
1. **Оркестратор** принимает запрос и понимает, что здесь нужен и агент карьеры, и агент обучения
2. **Агент карьеры** лезет в HR-данные: смотрит текущий грейд, KPI, результаты 360°
3. **Агент обучения** на основе gap-анализа подбирает конкретные курсы из библиотеки
4. Оркестратор собирает ответы обоих агентов и возвращает сотруднику готовый персональный план

Человек получает конкретный ответ — без ручного сбора данных из пяти систем.

---

## Агенты системы

| Агент | Что делает | Откуда берёт данные |
|---|---|---|
| **Оркестратор** | Принимает запрос, роутит к нужным агентам, собирает финальный ответ | — |
| **Агент обучения** | Подбирает курсы, строит персональную траекторию | Каталог курсов (LMS), грейд, роль сотрудника |
| **Агент онбординга** | Ведёт новичка по шагам адаптации, отвечает на вопросы | База знаний компании, регламенты, FAQ |
| **Агент карьеры** | Анализирует профиль и строит карьерный трек | KPI, оценки 360°, грейды, целевые роли |
| **Агент аналитики** | Показывает менеджеру прогресс команды, флаги отставания | Данные о прохождении курсов, KPI команды |

## Ключевая техника: RAG

Каждый агент не просто спрашивает у LLM — он сначала ищет релевантные данные в корпоративной базе знаний и подкладывает их в контекст запроса. Это называется **RAG (Retrieval-Augmented Generation)**.

Пример для агента онбординга:
1. Новичок спрашивает: «Как оформить командировку?»
2. Агент ищет в Qdrant документы про командировки
3. Находит нужный регламент и кладёт его в контекст LLM
4. LLM отвечает конкретно и точно, а не галлюцинирует

Без RAG агент отвечал бы наугад. С RAG — он знает только то, что реально есть в компании.

## Стек

### Бэкенд
- **Python + FastAPI** — основной язык и API-сервер
- **LangGraph** — фреймворк для построения агентов и оркестрации (граф состояний, удобно отлаживать)
- **LangChain** — утилиты для работы с LLM, загрузки документов, RAG-пайплайна

### LLM (языковая модель)
- **GigaChat Pro (Сбер)** — российская модель, важно для корпоративного клиента с точки зрения данных и санкций
- Резервно: **YandexGPT** или self-hosted **Mistral/LLaMA** на GPU-сервере (если данные нельзя отправлять во вне)

### База данных
- **PostgreSQL** — основное хранилище (профили сотрудников, курсы, KPI, прогресс)
- **Qdrant** — векторная база для RAG (хранит эмбеддинги документов, регламентов, курсов)
- **Redis** — кэш и очереди задач между агентами

### Фронтенд
- **Next.js (React)** — портал сотрудника и дашборд менеджера
- **TypeScript** — типизация
- **Tailwind CSS** — стили

### Инфраструктура
- **Docker + Docker Compose** — контейнеризация всего стека
- **Kubernetes** — для продакшна (масштабирование агентов под нагрузку)
- **Nginx** — реверс-прокси
- Деплой: **on-premise или частное облако** (критично для Газпром Нефти — данные сотрудников не должны уходить во внешние облака)

### Интеграции
- HR-система компании (API или экспорт данных) — для получения KPI, грейдов, 360°
- LMS (существующая библиотека курсов) — каталог и прогресс обучения
- Корпоративный SSO (Active Directory / Keycloak) — авторизация через корп. аккаунт

## Поток запроса (упрощённо)

```
Сотрудник вводит запрос
        ↓
FastAPI принимает запрос, проверяет авторизацию
        ↓
LangGraph Orchestrator анализирует намерение
        ↓
    [параллельно или последовательно]
   Агент 1          Агент 2
  делает RAG       делает RAG
  запрос в БД      запрос в БД
        ↓                ↓
   возвращает        возвращает
   результат         результат
        ↓
Оркестратор собирает итоговый ответ
        ↓
Фронтенд показывает результат сотруднику
```

## Что важно сказать на защите

1. **Данные не покидают контур компании** — LLM либо российская (GigaChat), либо self-hosted. Это принципиально для Газпром Нефти.

2. **Агенты независимы** — если один агент падает, остальные работают. Система устойчива.

3. **Система обучается** — чем больше сотрудников пользуются, тем лучше становятся рекомендации (feedback loop через логи взаимодействий).

4. **Интеграция без замены** — система не ломает существующие HR-инструменты, она читает из них данные через API.
