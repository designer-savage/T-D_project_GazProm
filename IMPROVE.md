## 1. Реальный LLM-стриминг

**Проблема:** `_split_chunks()` в `orchestrator.py` режет готовый ответ по 20 слов — это имитация. Пауза перед появлением первого слова заметна.

**Решение:** Заменить `llm.ainvoke()` на `llm.astream()` во всех агентах, пробрасывать токены напрямую в SSE.

```python
# Сейчас в агентах:
response = await llm.ainvoke(prompt)
return response.content

# Нужно — агент становится async generator:
async def run(query, employee_context) -> AsyncGenerator[str, None]:
    async for chunk in llm.astream(prompt):
        yield chunk.content
```

В `orchestrator.py` убрать `_split_chunks()`, итерировать по генератору агента напрямую.

## 2. LLM-классификация намерений

**Проблема:** `_classify_intent()` в `orchestrator.py` — keyword matching по 20 словам. Запрос «хочу расти как специалист» или «что мне сделать для повышения зарплаты» уходит в `mixed` по дефолту вместо `career`.

**Как у лидеров:** SAP Joule и Sana используют intent recognition через LLM с контекстом пользователя и активных приложений.

**Решение:** Один быстрый LLM-вызов перед роутингом. Промпт описывает три агента, модель возвращает JSON с выбором.

```python
CLASSIFY_PROMPT = """Определи намерение запроса сотрудника. Выбери одно из: career, learning, onboarding, mixed.

career — карьерный рост, грейд, повышение, переход в новую роль, развитие как специалиста
learning — обучение, курсы, навыки, что изучить, как прокачать конкретное умение
onboarding — процессы компании, политики, как что-то оформить, командировки, доступы
mixed — запрос одновременно про карьеру и обучение, или тема не определена

Запрос: "{query}"
Ответь одним словом."""
```

Убрать `_classify_intent()`, заменить вызовом LLM. Groq отвечает за ~200 мс, задержка незаметна.

## 3. Skill Gap Analysis

**Проблема:** Агент карьеры видит компетенции сотрудника, но не знает что нужно для следующего грейда. Рекомендации получаются общими.

**Как у лидеров:** Cornerstone (55k навыков) и Degreed строят весь пайплайн вокруг gap: текущий уровень → целевой уровень роли → конкретный курс для закрытия.

**Решение:**

1. Добавить таблицу `role_requirements` в `db/schema.sql`:

```sql
CREATE TABLE role_requirements (
    grade TEXT NOT NULL,
    skill_name TEXT NOT NULL,
    required_level INTEGER NOT NULL,
    PRIMARY KEY (grade, skill_name)
);
```

2. Заполнить в `core/seed_data.py` — минимальные уровни навыков для каждого грейда (junior/middle/senior/lead).

3. В `career_agent.py` добавить запрос, который JOIN-ит `competencies` с `role_requirements` для следующего грейда и считает реальный gap.

4. Передавать в промпт не просто "компетенции сотрудника", а "что конкретно не дотягивает до следующего уровня".

## 4. Team Readiness Dashboard

**Проблема:** `/dashboard` показывает общую статистику — цифры без смысла для менеджера.

**Как у лидеров:** Cornerstone, Oracle и все топ-6 дают менеджеру карту готовности команды — кто где, кто готов к повышению, у кого критичные gap-ы.

**Решение:** Новый endpoint `GET /dashboard/team-readiness?manager_id=mgr_001` + компонент на фронте.

Что показывать:
- Таблица сотрудников с прогресс-барами по ключевым компетенциям
- Флаги: кто готов к переходу на следующий грейд (gap < порога), кто отстаёт
- Средний KPI команды по периодам

Данные уже есть в БД — нужны только SQL-запросы и UI.

## 5. Семантический поиск (улучшенный RAG)

**Проблема:** `rag/retriever.py` использует FTS5 — поиск по совпадению слов. «Командировочные расходы» не найдёт документ с «расходы на служебные поездки».

**Как у лидеров:** Workday/Sana и Docebo Neural Search используют векторный поиск — поиск по смыслу, а не по словам.

**Решение:** Добавить `chromadb` или `sqlite-vec` + `sentence-transformers` (модель `paraphrase-multilingual-MiniLM-L12-v2`, работает офлайн).

Схема изменений:
- `rag/embedder.py` — при старте индексирует `knowledge_documents` в векторное хранилище
- `rag/retriever.py` — hybrid search: сначала векторный (семантика), FTS5 как fallback
- Переиндексация при `SEED_DB=true`

Размер модели ~120 МБ, инференс на CPU за ~50 мс — приемлемо для демо.

## 6. Карьерные треки с целевыми ролями

**Проблема:** Сотрудник не видит "дорожную карту" — что конкретно произойдёт если он дорастёт до следующего грейда: какие роли открываются, какие задачи, какой уровень зарплатной вилки.

**Как у лидеров:** SAP Opportunity Marketplace и Oracle Job Discovery Agent показывают внутренние вакансии и роли, матчат с профилем.

**Решение:** Страница `/career` вместо статики — динамический трек на основе текущего грейда. Визуализация: текущий грейд → следующий → через один, с gap-ами на каждом переходе. Данные из `role_requirements` (п. 3).

## 7. Контекстная инъекция в промпт

**Проблема:** Агенты получают данные сотрудника, но не получают контекст сессии — о чём шёл разговор до этого запроса.

**Как у лидеров:** SAP Joule обогащает каждый промпт контекстом: в каком приложении находится пользователь, что делал последние 10 минут, какие данные открыты.

**Решение:** Передавать в `POST /chat/stream` последние N сообщений чата. В оркестраторе добавлять их в промпт как `История диалога`. Это сделает follow-up вопросы осмысленными — сейчас каждый запрос изолирован.

```python
# В теле запроса добавить:
{
  "query": "...",
  "employee_id": "emp_001",
  "history": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

## 8. Агент-уточнитель

**Проблема:** При неоднозначном запросе система уходит в `mixed` и запускает оба агента — даже если пользователь хотел только один ответ.

**Как у лидеров:** Degreed Maestro ведёт диалог перед тем как дать рекомендацию — уточняет контекст, чтобы ответ был точнее.

**Решение:** Если LLM-классификатор (п. 2) не уверен в намерении (вернул `mixed` или добавить confidence score), вместо запуска обоих агентов — SSE-событие `clarification` с уточняющим вопросом. Пользователь отвечает, запрос роутится точно.

Это сложнее реализовать (нужен stateful диалог), поэтому в конце очереди.
