# Улучшения прототипа

Пять точечных доработок, которые делают демо цельным без раздувания стека. Все реализуемы на выходных, не требуют новых зависимостей.

---

## 1. Реальный LLM-стриминг

**Проблема:** `_split_chunks()` в `orchestrator.py` режет готовый ответ по 20 слов — это имитация. Пауза перед первым словом заметна.

**Решение:** Заменить `llm.ainvoke()` на `llm.astream()` во всех агентах, пробрасывать токены напрямую в SSE.

```python
# Сейчас:
response = await llm.ainvoke(prompt)
return response.content

# Нужно — агент становится async generator:
async def run(query, employee_context) -> AsyncGenerator[str, None]:
    async for chunk in llm.astream(prompt):
        yield chunk.content
```

Убрать `_split_chunks()` из оркестратора, итерировать по генератору агента напрямую.

## 2. LLM-классификация намерений

**Проблема:** `_classify_intent()` — keyword matching по 20 словам. «Хочу расти как специалист» уходит в `mixed` вместо `career`.

**Решение:** Один быстрый LLM-вызов перед роутингом. Groq отвечает за ~200 мс, задержка незаметна.

```python
CLASSIFY_PROMPT = """Определи намерение запроса. Выбери одно: career, learning, onboarding, mixed.

career — карьерный рост, грейд, повышение, развитие как специалиста
learning — обучение, курсы, навыки, что изучить
onboarding — процессы компании, политики, как что-то оформить
mixed — одновременно про карьеру и обучение, или тема не определена

Запрос: "{query}"
Ответь одним словом."""
```

## 3. История диалога в промпте

**Проблема:** Каждый запрос изолирован — агент не знает, о чём шёл разговор. Follow-up вопросы («а что насчёт Python?») непонятны без контекста.

**Решение:** Добавить `history` в тело запроса и пробрасывать в промпт.

```python
# POST /chat/stream
{
  "query": "...",
  "employee_id": "emp_001",
  "history": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

Фронтенд: при монтировании чата — `GET /chat/history?employee_id=...&limit=10`, инициализировать стейт. Бесшовный переход при переключении роли и обновлении страницы.

## 4. Reviewer Agent — минимальная проверка качества

**Проблема:** Если llama-3.3-70b вернул короткий или пустой ответ, система молча отдаёт его пользователю.

**Решение:** `agents/reviewer.py` — постпроцессор без LLM-вызова (только эвристика, нулевая латентность).

Два условия для отклонения ответа:
1. Длина < 100 токенов
2. В ответе нет ни одного факта из RAG-чанков

При выполнении хотя бы одного — оркестратор отправляет SSE-событие `clarification` с уточняющим вопросом вместо пустого ответа.

## 5. At-Risk флаг в дашборде менеджера

**Проблема:** Менеджер видит цифры, но не знает, кому нужна помощь прямо сейчас.

**Решение:** Один SQL-запрос в `routers/dashboard.py`, флаг в UI — данные уже есть в БД.

```sql
SELECT e.employee_id, e.name,
       COUNT(lp.course_id) AS total,
       SUM(CASE WHEN lp.progress_pct < 50 THEN 1 ELSE 0 END) AS behind
FROM employees e
JOIN learning_progress lp ON lp.employee_id = e.employee_id
WHERE lp.assigned_date >= date('now', '-1 month')
GROUP BY e.employee_id
HAVING (behind * 1.0 / total) > 0.5
```

Сотрудники из выборки подсвечиваются в таблице команды как `at-risk`.
